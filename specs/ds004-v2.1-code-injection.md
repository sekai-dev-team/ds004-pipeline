# DS-004 v2.1: Code-Injected Frontmatter & Related Articles

> **Status:** Draft  
> **Date:** 2026-06-01  
> **Parent:** DS-004 Phase 1 (Semantic Consolidation Pipeline)  
> **Root Cause:** LLM is unreliable at generating structured data (frontmatter) and comprehensive cross-references (related articles). These MUST be generated by deterministic code, not probabilistic LLM output.

---

## Problem Statement

After running DS-004 against 48 episodic notes, the output has two systemic failures:

### 1. Missing Frontmatter (16/32 concept pages)

| Status | Count | Examples |
|--------|-------|----------|
| Has frontmatter | 16 | minimax-m3, factual-density-in-rag |
| Missing frontmatter | 16 | dyna-tree, agent-security, hermes-agent, claude-code-ultracode |

The LLM prompt asks for `full_page_content` including YAML frontmatter, but the LLM is inconsistent. Half the time it omits `---` delimiters entirely.

### 2. Insufficient / Wrong Related Articles

| Concept | Wikilinks | Issue |
|---------|-----------|-------|
| minimax-m3 | 1 (self-ref) | Only references itself |
| factual-density-in-rag | 1 | Should have 3+ related articles from arXiv |
| gpu-kernel-surrogate | 1 | Same paper family as other GPU/optimization papers |
| agent-threat-bench | 1 | Should cross-link agent-security, stateful-monitoring |
| Most concepts | ≤1 | LLM only sees 5 matched notes in context |

The LLM only receives `matched_notes[:5]` as search context and is told to write the "## 相关文章" section. It cannot discover all related articles because:
- k-mcp search returns broader results than what the LLM sees
- The LLM's context window limits how many search results we can pass

---

## Design Decision: Split Code vs LLM Responsibility

```
                    CODE (deterministic)              LLM (probabilistic)
                    ─────────────────────              ────────────────────
Frontmatter         ✅ Generate & inject               ❌ Never touch
Related articles    ✅ Search k-mcp, format wikilinks   ❌ Never touch
Concept synthesis   ❌ Never touch                     ✅ Generate body text
Index.md            ✅ Inject (list-based)             ❌ Never touch
Log.md              ✅ Inject (structured)             ❌ Never touch
```

The LLM now receives a clean prompt: "Here is the new note, here are matched notes for context, here is the existing concept page. Generate ONLY the concept body (no frontmatter, no 相关文章 section)."

## Implementation Checklist

### Phase A: Frontmatter Injection (code-side)

- [ ] **A1.** Create `_inject_frontmatter(name, title, body_content, related_paths, date_str) -> str`
  - Generates complete markdown with frontmatter + body
  - Frontmatter MUST include:
    ```yaml
    ---
    memory_type: semantic
    concept: {name}
    title: "{title}"
    tags: [type/semantic]
    created: {date_str}
    last_updated: {date_str}
    related_count: {n}
    ---
    ```
  - `created` is preserved from existing page on updates (read existing frontmatter first)
  - `related_count` is the number of related article wikilinks inserted
  - Do NOT put wikilinks in frontmatter — they go in "## 相关文章" body section

- [ ] **A2.** Modify `_concept_page_template()` to accept `related_paths: list[str]`
  - Generate the "## 相关文章" section with wikilinks:
    ```markdown
    ## 相关文章
    
    - [[Evaluating_Factual_Density_in_Multi-Source_RAG__A_Study_in_Medical_AI_Accuracy]]
    - [[GPU_Forecasters__Language_Models_as_Selective_Surrogates_for_Kernel_Runtime_Optimization]]
    - [[Semantic_Triplet_Restoration__A_Novel_Protocol_for_Hierarchical_Table_Understanding]]
    ```
  - Remove "## 核心子主题" and "## 待探索问题" from template — let LLM generate these organically

- [ ] **A3.** Modify `consolidate()` to inject frontmatter after LLM returns
  - Step 5 calls LLM → LLM returns body-only markdown (no `---`, no "## 相关文章")
  - Step 7 wraps LLM body with `_inject_frontmatter()` before writing
  - On UPDATE: read existing concept page's frontmatter, preserve `created`, update `last_updated`
  - On CREATE: set `created` to today

### Phase B: Related Articles Injection (code-side)

- [ ] **B1.** Create `_find_all_related(new_note_path, content, limit=15) -> list[dict]`
  - Calls k-mcp `search()` with query extracted from note content
  - Filters: exclude self, exclude non-episodic (concepts/, digests/, insights/)
  - Returns up to `limit` results with `path` and `vec_score`
  - Logs the count and top scores

- [ ] **B2.** Modify trigger logic: separate "trigger threshold" from "related article injection"
  - **Trigger decision**: requires `MATCH_THRESHOLD` (2) high-quality matches (vec_score ≥ 0.75) → this gates whether consolidation happens at all
  - **Related article injection**: always inject ALL matches (up to 15) regardless of trigger threshold — because once trigger is met, we want comprehensive cross-linking

- [ ] **B3.** Modify `consolidate()` to inject related wikilinks
  - After LLM returns body, append "## 相关文章" section with ALL related article paths as `[[wikilinks]]`
  - This happens BEFORE `_inject_frontmatter()` wraps the final content
  - Related articles are sorted by vec_score descending

### Phase C: LLM Prompt Cleanup

- [ ] **C1.** Rewrite system prompt for body-only output
  - LLM generates ONLY the markdown body (no frontmatter, no "## 相关文章")
  - Output format changes from `full_page_content` to `body_content`
  - System prompt explicitly states: "Do NOT include YAML frontmatter (---). Do NOT include a '## 相关文章' section. These will be injected by the pipeline automatically."
  - LLM is now ONLY responsible for: 概述, 核心观点, 趋势判断, ⚠️ 矛盾标注

- [ ] **C2.** Update JSON output schema
  ```json
  {
    "concepts_to_update": [
      {
        "name": "kebab-case-concept-name",
        "title": "Human-readable title",
        "action": "update" or "create",
        "body_content": "Markdown body ONLY — no frontmatter, no 相关文章"
      }
    ],
    "index_update": "Content to append to index.md",
    "log_entry": "Single line for log.md"
  }
  ```

### Phase D: Validation & Safety

- [ ] **D1.** Add `_validate_frontmatter(content) -> bool` — called before writing any concept page
  - Verifies content starts with `---`
  - Verifies required fields: `memory_type`, `concept`, `created`, `last_updated`
  - Logs warning and falls back to template if validation fails

- [ ] **D2.** Add `_validate_related_section(content) -> bool` — verifies "## 相关文章" exists and has ≥1 wikilink
  - Logs warning if missing

- [ ] **D3.** When LLM `body_content` is empty or invalid, write a minimal concept page (template + frontmatter + "## 概述\n\nLLM consolidation failed. Related articles linked below." + related articles section)

### Phase E: Side Effects

- [ ] **E1.** `_parse_llm_json()` must still work — the JSON key changed from `full_page_content` to `body_content`
- [ ] **E2.** `consolidate_all()` summary must include counts for: frontmatter_validated, frontmatter_fallback, related_injected
- [ ] **E3.** Test: run against the current vault, verify all 32 concept pages have:
  - Valid YAML frontmatter with all required fields
  - "## 相关文章" section with ≥2 wikilinks (except genuinely isolated concepts)
- [ ] **E4.** Re-run trigger for concepts that had low related count — some may now meet threshold with improved search
